# AI Story Generator 하네스 엔지니어링 설계서

## 1. 설계 개요

본 문서는 OpenAI의 하네스 엔지니어링 원칙을 `AI Story Generator` 프로젝트의 Codex 에이전트 운영 방식에 맞게 적용하기 위한 설계안이다.

현재 프로젝트 지침서인 `AGENT_AI_Story_Dev_Report_Only.md`는 Codex 에이전트의 역할을 개발 기능 구현이 아닌 Markdown 개발 보고서 작성으로 제한한다. 따라서 본 설계의 목적은 에이전트가 코드를 직접 구현하도록 만드는 것이 아니라, 설치, 빌드, 실행, Docker 이미지화, API 검증, 오류 해결 과정을 읽기 쉽고 검증 가능한 보고서로 작성하도록 지원하는 하네스를 정의하는 것이다.

참고 원문:

- OpenAI, 「하네스 엔지니어링: 에이전트 우선 세계에서 Codex 활용하기」, 2026-02-11
- https://openai.com/ko-KR/index/harness-engineering/

## 2. 적용 원칙

### 2.1 사람은 조정하고 에이전트는 수행한다

사람은 목표, 수용 기준, 우선순위, 검증 기준을 제공한다. Codex 에이전트는 로컬 저장소의 문서와 확인 가능한 실행 결과를 근거로 보고서를 작성한다.

본 프로젝트에서 사람의 역할은 다음과 같다.

- 작업 목표 지정
- 실제 개발자가 수행한 명령어와 로그 제공
- 불명확한 실행 결과 판단
- 최종 보고서 승인

Codex 에이전트의 역할은 다음과 같다.

- 지침서와 기존 Markdown 문서 확인
- 설치, 빌드, 실행, 검증 과정을 보고서 목차에 맞게 정리
- 오류 발생 시 원인, 해결 방법, 재검증 결과를 분리하여 기록
- 확인되지 않은 성공 결과를 성공으로 단정하지 않음

### 2.2 AGENT 문서는 목차 역할을 한다

OpenAI 원문은 거대한 단일 지침 파일보다 짧은 진입점과 구조화된 지식 저장소를 권장한다. 현재 프로젝트의 `AGENT_AI_Story_Dev_Report_Only.md`는 상세 지침을 많이 포함하고 있으므로, 장기적으로는 다음과 같이 분리한다.

```text
AGENT_AI_Story_Dev_Report_Only.md
docs/
├── index.md
├── reporting_rules.md
├── architecture_summary.md
├── verification_rules.md
├── docker_reporting_rules.md
└── api_test_reporting_rules.md
reports/
├── development_report.md
├── install_report.md
├── build_report.md
├── docker_report.md
├── api_test_report.md
├── error_resolution_report.md
└── history/
```

단기적으로는 기존 지침서를 유지하되, 새 규칙을 추가할 때 전체 지침서를 계속 키우지 않고 `docs/` 하위 문서로 분리하는 방식을 권장한다.

### 2.3 에이전트가 읽을 수 있는 것만 신뢰한다

에이전트는 저장소에 존재하는 Markdown 문서, 명령어 기록, 로그 발췌, 보고서만 근거로 삼는다. 채팅, 구두 설명, 외부 문서에만 있는 정보는 보고서의 확정 근거로 사용하지 않는다.

보고서에 포함 가능한 근거는 다음과 같다.

- 실행한 명령어
- 실행 결과 원문 또는 핵심 로그
- 변경 또는 생성된 파일 경로
- 오류 메시지
- 재검증 결과
- 추가 확인 필요 사항

보고서에 포함하면 안 되는 내용은 다음과 같다.

- 실제 확인하지 않은 성공 결과
- 존재하지 않는 파일 변경 내역
- 구현하지 않은 기능의 완료 표현
- Markdown 외 산출물 생성 계획

## 3. 하네스 구성

### 3.1 목표 입력 하네스

목표 입력 하네스는 사람이 Codex에 작업을 요청할 때 필요한 최소 입력을 표준화한다.

```md
## 작업 요청 입력

- 작업명:
- 대상 보고서:
- 실제 수행한 명령어:
- 실행 환경:
- 성공 로그:
- 오류 로그:
- 추가 확인 필요 사항:
```

이 입력이 불완전하면 Codex는 성공으로 단정하지 않고 `추가 확인 필요 사항`을 보고서에 남긴다.

### 3.2 지식 맵 하네스

지식 맵 하네스는 에이전트가 어떤 문서를 먼저 읽고 어떤 규칙을 우선 적용해야 하는지 정의한다.

| 우선순위 | 문서 | 용도 |
|---|---|---|
| 1 | `AGENT_AI_Story_Dev_Report_Only.md` | 에이전트 역할, 금지 작업, 보고서 형식 확인 |
| 2 | `reports/development_report.md` | 통합 개발 과정 확인 |
| 3 | `reports/install_report.md` | 설치 과정 확인 |
| 4 | `reports/build_report.md` | 빌드 과정 확인 |
| 5 | `reports/docker_report.md` | Docker 이미지화 및 실행 과정 확인 |
| 6 | `reports/api_test_report.md` | API 검증 과정 확인 |
| 7 | `reports/error_resolution_report.md` | 오류 원인 및 해결 이력 확인 |

### 3.3 보고서 작성 하네스

보고서 작성은 다음 순서로 수행한다.

1. 대상 보고서 종류 확인
2. 기존 보고서가 있으면 구조 유지
3. 작업 단위별 명령어, 결과, 판단 분리
4. 오류가 있으면 원인, 해결 방법, 재검증 결과 분리
5. 확인되지 않은 항목은 `확인되지 않음`으로 명시
6. Markdown 문법과 파일 경로 확인

보고서의 기본 단위는 다음 형식을 따른다.

```md
## TASK-번호: 작업명

### 1. 작업 목적

### 2. 작업 전 상태

### 3. 수행 명령어

### 4. 변경 또는 생성된 파일

### 5. 실행 결과

### 6. 오류 발생 여부

### 7. 해결 방법

### 8. 검증 결과

### 9. 다음 작업
```

### 3.4 검증 하네스

검증 하네스는 보고서 작성 후 반드시 확인해야 하는 조건을 정의한다.

| 검증 항목 | 기준 |
|---|---|
| Markdown 형식 | 산출물은 `.md` 파일이어야 함 |
| 저장 위치 | 기본 산출물은 `reports/` 하위에 있어야 함 |
| 금지 작업 준수 | FastAPI, Router, Service, Client, 모델 추론 코드 생성 금지 |
| 명령어 기록 | 실행 명령어는 코드 블록으로 기록해야 함 |
| 결과 기록 | 실행 결과 또는 확인 불가 상태를 명시해야 함 |
| 오류 추적 | 오류 메시지, 원인, 해결 방법, 재검증 결과를 분리해야 함 |
| 과장 금지 | 확인되지 않은 성능 수치나 완료 표현을 사용하지 않아야 함 |
| 재현성 | 다른 개발자가 동일 절차를 따라갈 수 있어야 함 |

### 3.5 리뷰 하네스

보고서 리뷰는 사람이 직접 모든 문장을 검토하는 방식보다 체크리스트 기반으로 수행한다.

```md
## 보고서 리뷰 체크리스트

- [ ] 대상 보고서 경로가 올바른가
- [ ] 실제 수행한 명령어와 결과가 분리되어 있는가
- [ ] 성공 여부가 근거 없이 단정되지 않았는가
- [ ] 오류가 있는 경우 원인과 해결 방법이 기록되어 있는가
- [ ] Markdown 외 산출물 생성 내용이 없는가
- [ ] API 내부 보고서 기능 생성 내용이 없는가
- [ ] 다음 작업 또는 추가 확인 필요 사항이 명시되어 있는가
```

## 4. 평가 하네스 설계

### 4.1 보고서 완성도 평가

각 보고서는 다음 기준으로 평가한다.

| 점검 항목 | 통과 기준 |
|---|---|
| 목차 적합성 | 지침서의 기본 목차를 유지함 |
| 절차 기록 | 작업 순서가 시간 흐름대로 기록됨 |
| 명령어 정확성 | 실행 명령어가 누락 없이 작성됨 |
| 결과 근거 | 로그, 상태, 파일 경로 등 확인 근거가 있음 |
| 오류 처리 | 오류 발생 여부와 해결 과정이 분리됨 |
| 추가 확인 | 불명확한 항목을 별도 표시함 |

### 4.2 Docker 보고서 평가

`reports/docker_report.md`는 다음 항목을 반드시 포함해야 한다.

- Dockerfile 작성 목적
- Base Image 선정 이유
- requirements.txt 설치 과정
- 소스 코드 복사 방식
- FastAPI 실행 명령어
- 포트 매핑 정보
- GPU 사용 옵션
- 볼륨 마운트 경로
- 이미지 빌드 명령어
- 컨테이너 실행 명령어
- 실행 검증 결과
- 발생 오류 및 해결 방법

### 4.3 API 테스트 보고서 평가

`reports/api_test_report.md`는 다음 흐름을 기준으로 평가한다.

1. FastAPI 서버 실행 확인
2. Swagger 문서 접근 확인
3. STT API 테스트
4. 동화 생성 API 테스트
5. Run 상태 조회 API 테스트
6. SSE 이벤트 스트림 테스트
7. 이미지 목록 조회 API 테스트
8. 오디오 목록 조회 API 테스트
9. 테스트 결과 요약
10. 오류 및 해결 내역

### 4.4 오류 해결 보고서 평가

`reports/error_resolution_report.md`는 오류를 단순히 나열하지 않고 다음 추적 구조를 유지해야 한다.

```text
발생 환경 -> 발생 명령어 -> 오류 메시지 -> 원인 분석 -> 해결 방법 -> 재검증 결과 -> 예방 방안
```

## 5. 에이전트 자율성 단계

본 프로젝트는 보고서 전용 에이전트를 전제로 하므로 자율성 단계도 문서화 중심으로 제한한다.

| 단계 | 설명 | 허용 작업 |
|---|---|---|
| L0 | 수동 기록 | 사람이 작성한 로그를 보고서 형식으로 정리 |
| L1 | 구조화 작성 | 기존 목차에 맞춰 보고서 초안 작성 |
| L2 | 근거 검증 | 명령어, 결과, 오류, 파일 경로의 누락 여부 점검 |
| L3 | 보고서 리뷰 | 체크리스트 기반으로 보고서 품질 검토 |
| L4 | 반복 개선 | 누락된 증거와 추가 확인 필요 사항을 다음 작업으로 정리 |

L4 단계에서도 Codex는 서비스 코드를 구현하지 않는다. 필요한 경우 개발자에게 추가 실행 로그 또는 파일 경로 확인을 요청하고, 확인된 결과만 보고서에 반영한다.

## 6. 운영 루프

하네스 운영 루프는 다음과 같다.

```text
작업 요청
-> 지침서 확인
-> 대상 보고서 선택
-> 실행 근거 수집
-> 보고서 작성
-> 검증 체크리스트 적용
-> 불명확한 항목 표시
-> 최종 Markdown 산출
-> 필요 시 history 보고서로 보관
```

작업별 판단 기준은 다음과 같다.

| 상황 | 처리 방식 |
|---|---|
| 실행 결과가 명확함 | 성공 또는 실패를 근거와 함께 기록 |
| 실행 결과가 없음 | `현재 실행 결과는 확인되지 않음`으로 기록 |
| 오류 원인이 불명확함 | 원인 후보와 추가 확인 필요 사항을 분리 |
| 구현 요청이 포함됨 | 구현하지 않고 보고서 작성 범위로 전환 |
| Markdown 외 산출물 요청 | 지침 위반 사항으로 보고하고 `.md` 산출물만 작성 |

## 7. 엔트로피 관리

에이전트가 생성하는 보고서는 시간이 지나며 중복, 오래된 명령어, 검증되지 않은 성공 표현이 누적될 수 있다. 이를 방지하기 위해 정기적인 문서 정리 기준을 둔다.

### 7.1 정리 대상

- 동일 명령어가 여러 보고서에 상충되게 기록된 경우
- 실제 결과 없이 성공으로 작성된 문장
- 더 이상 사용하지 않는 Docker 실행 옵션
- 파일 경로가 변경되었으나 이전 경로가 남아 있는 경우
- 오류 해결 보고서에 재검증 결과가 없는 경우

### 7.2 정리 방식

- 기존 보고서 본문을 삭제하지 않고 `정정 내역` 또는 `추가 확인 필요 사항`으로 보완
- 날짜별 이력이 필요한 경우 `reports/history/`에 별도 보관
- 확인되지 않은 과거 결과는 성공으로 유지하지 않고 확인 상태를 명시

## 8. 단계별 도입 계획

### 8.1 1단계: 문서 지식 맵 정리

- `AGENT_AI_Story_Dev_Report_Only.md`를 최상위 지침으로 유지
- 세부 규칙은 향후 `docs/` 하위 Markdown 문서로 분리
- 보고서 산출물은 계속 `reports/` 하위에 저장

### 8.2 2단계: 보고서 템플릿 고정

- 설치 보고서 템플릿 정리
- 빌드 보고서 템플릿 정리
- Docker 보고서 템플릿 정리
- API 테스트 보고서 템플릿 정리
- 오류 해결 보고서 템플릿 정리

### 8.3 3단계: 검증 체크리스트 적용

- 보고서 작성 후 체크리스트를 통과하지 못한 항목을 `추가 확인 필요 사항`으로 기록
- 성공, 실패, 미확인을 명확히 구분
- 검증되지 않은 결과를 완료로 표현하지 않음

### 8.4 4단계: 반복 정리 체계 운영

- 오래된 보고서와 최신 보고서의 경로, 명령어, 결과를 비교
- 상충되는 내용은 정정 내역으로 보완
- 반복 오류는 `reports/error_resolution_report.md`에 누적 정리

---

## 9. 최종 설계 기준

본 프로젝트의 하네스 엔지니어링은 다음 기준을 따른다.

- Codex 에이전트의 핵심 역할은 코드 구현이 아니라 보고서 작성이다.
- 하네스는 에이전트가 정확한 Markdown 보고서를 작성하도록 돕는 구조이다.
- 저장소 안에 기록되지 않은 정보는 확정 근거로 사용하지 않는다.
- 실행 결과가 없으면 성공으로 단정하지 않는다.
- 보고서는 재현성, 명확성, 검증성, 추적성, 간결성을 만족해야 한다.
- 문서 구조는 점진적으로 분리하되, 최종 산출물은 `reports/` 하위 Markdown 파일로 유지한다.

## 10. 결론

AI Story Generator의 하네스 엔지니어링은 에이전트에게 더 많은 구현 권한을 부여하는 설계가 아니라, 제한된 역할 안에서 더 정확하고 검증 가능한 보고서를 작성하도록 만드는 운영 설계이다.

OpenAI 원문의 핵심인 환경 설계, 의도 명시, 피드백 루프, 구조화된 지식 저장소, 에이전트 가독성, 불변 조건 강제 원칙을 본 프로젝트에 적용하면 다음 방향이 된다.

- `AGENT_AI_Story_Dev_Report_Only.md`는 최상위 작업 지침으로 유지
- 세부 지식은 향후 `docs/` 하위 Markdown으로 분리
- 실제 산출물은 `reports/` 하위 Markdown 보고서로 관리
- 검증 루프는 명령어, 결과, 오류, 재검증 중심으로 운영
- 에이전트 자율성은 보고서 작성과 보고서 품질 검토 범위로 제한

이 설계를 따르면 Codex 에이전트는 프로젝트의 구현 경계를 넘지 않으면서도, 개발 과정의 설치, 빌드, 실행, Docker, API 검증, 오류 해결 이력을 일관된 품질로 축적할 수 있다.
